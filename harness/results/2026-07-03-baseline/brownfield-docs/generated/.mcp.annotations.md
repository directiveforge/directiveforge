# MCP & Config Annotations — Trailhead Collective

> Companion to `.claude/settings.json`. There is deliberately **no `.mcp.json`** in this repo.
> Under governance decision **D4**, no external service or credential is wired up without a
> founder bootstrap — and an MCP server is an external integration. So MCP here is
> **propose-only**: candidates are recorded below as proposals; nothing is connected until the
> founder bootstraps it and moves it into `governance/INTEGRATIONS-REGISTRY.md`.

## Why no MCP servers are configured

1. **Governance (D4)** — the single most important reason. Even a server that passes every
   quality gate stops at a proposal here. See `governance/INTEGRATIONS-REGISTRY.md`.
2. **Fit** — this is a pure-Markdown ops corpus. There is no database, no payments code, no
   framework whose APIs an agent must look up. The realistic MCP surface is tiny.

## Candidate servers (proposals, NOT wired)

### github — remote HTTP (`https://api.githubcopilot.com/mcp/`)

- **Posture**: RW (repos, issues, PRs) via OAuth browser consent; no PAT env var to store.
- **Status**: PROPOSE-ONLY. Applies only if/when this repo is hosted on GitHub (it currently
  has no git remote). If the founder puts it on GitHub and wants issue/PR access on the docs,
  draft a registry proposal row, then the founder bootstraps the OAuth consent. Until then,
  the `gh` CLI via Bash covers any occasional need.
- **Registry gate**: not yet vetted in `governance/INTEGRATIONS-REGISTRY.md`.

### context7 — documentation lookup MCP

- **Posture**: RO — fetches public library docs; no credentials.
- **Status**: NOT APPLICABLE. The corpus discusses no library or framework APIs, so there is
  nothing for Context7 to ground. Reconsider only if the repo ever documents software.

### Database / payments / CMS connectors

- **Status**: NOT APPLICABLE. No database, no runtime, no code. DepotHold Payments and
  Ledgerline Books are vendor accounts governed through the integrations registry, not MCP
  targets — and any access to them stays a founder-bootstrapped credential (D4). Never route
  customer card/PII data through an agent surface.

## If the founder wants to add a server later

1. Confirm the server passes the kit's 5 quality gates (EXISTS / MAINTAINED / TRUSTED /
   SECURE via `uvx snyk-agent-scan@latest` / USEFUL).
2. Draft a proposal row in `governance/INTEGRATIONS-REGISTRY.md` under "Proposed — awaiting
   founder bootstrap": tool, job, why existing tools can't cover it, recurring cost, and the
   least-privilege credential surface.
3. The **founder** creates `.mcp.json`, pins the version (`@x.y.z`), uses `${ENV_VAR}` for any
   credential, and moves the registry row to "Vetted and in use." An assistant never does this.

## Appendix — `.claude/settings.json` levers

Semantics per `<KIT_ROOT>/workflows/WORKFLOW-CLAUDE-CODE.md` (env vars verified 2026-07-02):

- `MAX_THINKING_TOKENS` — set to `10000` here (routine doc work rarely needs deep thinking).
- `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` — `50` for long freshness-audit sessions; compacts before the quality cliff.
- `CLAUDE_CODE_SUBAGENT_MODEL` — **omitted** (inherit, v2.1.196+).
- `hooks` — `{}` (no formatter; there is no code toolchain to format Markdown against in this repo).
- Permissions: pre-allow safe read-only corpus commands (git reads, `grep`, `find`, `wc`, `ls`, `cat`); deny destructive (`rm -rf`, force-push, hard reset) and `.env` reads.
