# SlotHarbor — Quality Gates

> Scope: whole project; the verification chain for every change.
> This repo ships NO test/lint/typecheck tooling (no pytest / ruff / black / mypy in `pyproject.toml`, no Makefile). The gates below are the checks that genuinely exist. If you add real tooling, add its row here and to `CLAUDE.md` Definition of Done.

## Gate table
| Gate | Command | Blocking |
|------|---------|----------|
| App boots | `python -c "import app.main"` (or `uvicorn app.main:app` starts clean) | Yes |
| Migrations apply | `alembic upgrade head` (against a dev DB) | Yes — when a migration changed |
| Migration review | Read the new file in `migrations/versions/` before applying | Yes — when autogenerated |

## When to run
- After every edit batch touching `app/`: import-boot the app
- After any model or migration change: `alembic upgrade head` on a dev/SQLite DB, and READ the generated migration first
- Before every commit: app boots clean

## Failure protocol
- A failing gate BLOCKS the commit — fix forward or revert; never bypass (no `--no-verify`)
- Report the exact command + exit code — "should work" is not a verification
- Alembic autogenerate misses column-type changes and some constraints — never trust it blind; the review gate is not optional

## Content policy
- Zero placeholder content (lorem, stub secrets, TODO copy) in any shipped surface

## Missing-tooling note
- There is no CI here and no automated test suite. Manual endpoint exercise is the last line of defense — actually call the changed route (curl / HTTPie / `/docs`) and confirm the response shape and status code before declaring done.

## Division of labor
- The verifier agent (`.claude/agents/verifier.md`) runs the boot + migration gates mechanically; this rule binds the main thread to the same gates.
