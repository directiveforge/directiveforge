## What surprised me

The single most repeated, independently-arrived-at diagnosis across this entire beat — Ronacher, Danjou, Hashimoto/Register, DORA, and Yegge's own Gas Town postmortem — is that **review, not code generation, is now the bottleneck**. Nobody needed to read each other's posts to converge on this; it shows up as "shooting up bad code is rude" (Ronacher), "the code review bottleneck is you" (Danjou), a measured 7.2% stability drop and 91%-ish qualitative "babysitting" complaints (DORA), and a maintainer literally pulling a 50k-star project off GitHub because PR load broke the platform (Hashimoto/Register). That's a genuinely convergent, cross-source signal, not cherry-picking.

Second surprise: named practitioners are far more anti-ceremony than I expected going in. I assumed people building serious agentic-engineering practice would be pro-process, pro-documentation, pro-rules-files — the natural audience for a workflow-kit pitch. Instead the most quotable reaction to an elaborate markdown-based agent-memory system (Beads, which is structurally similar to what a rules/skills generator ships) was Ronacher calling it "240,000 lines of code... abysmal" code quality and the surrounding culture "a Mad Max cult... insane psychosis." Hamel Husain independently argues the minimum viable eval practice is "30 minutes" of manual review, not tooling, and warns against delegating that judgment away. This is real, load-bearing disconfirming evidence: the same people who'd be the ideal early adopters of a workflow-generator are primed to mock exactly this category of artifact if it doesn't demonstrate measured value fast.

Third: the human/physical cost angle (Yegge's "vampiric effect," daytime fatigue, a self-imposed 3-hour focus cap) never came up in my initial framing of "context, trust, review, cost, instruction drift." It's a real, recurring theme in his account and worth carrying forward — pain isn't only architectural, it's also literally exhausting for the people driving the loop.

Fourth: Yegge's own Gas Town is simultaneously the best evidence *for* harness/rules engineering (it "just works" now and people "build real stuff" on it) and the best cautionary tale *against* assuming a memory/rules system works out of the box — it took months of production data loss and a full storage-layer rewrite (SQLite+JSONL → Dolt) before it stabilized. A generator that ships rules/skills/memory scaffolding on day one is making an implicit promise ("this part is solved for you") that the most sophisticated builder in this space needed months of real production pain to actually solve.

Fifth: infrastructure-layer pain (GitHub reliability, merge-queue contention, Git's ref storage under agent-scale churn) is a distinct, verifiable pain category almost entirely outside what any workflow-kit, rules file, or per-repo harness can touch — it's a shared-platform capacity problem. Three independent named sources (Hashimoto, Chacon, Danjou) converged on this same layer within weeks of each other in 2026.

Sixth: the strongest, most repeatedly-praised pattern across every source that has *any* positive things to say — Hashimoto's "harness engineering" (build a lint/test the agent can self-invoke instead of manual correction), Eugene Yan's "shift verification left," Danjou's speculative/batched merge-queue fixes, StrongDM's independent scenario-testing holdout sets — is convergent, cross-source, and squarely "verification-first, not ceremony-first." That's the actual validated wedge, if a kit can show the verification loop working and measured, rather than asserting rigor via documentation volume.
