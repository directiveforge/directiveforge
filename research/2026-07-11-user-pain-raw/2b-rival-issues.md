## What surprised me

1. The single highest-signal complaint across all five repos by far — spec-kit #1191 at 105 upvotes / 114 total reactions — is not an install bug or a token complaint, it's "I built a spec and now I can't cleanly change it." Every rival kit that generates durable artifacts (specs, epics, plans, story files) eventually has to answer "what happens on version 2 of this artifact," and none of the five repos I looked at have shipped a clean answer. BMAD has the same wound from a different angle (#2527/#2553, not in the main table but visible in the recent-issues dump: "overwrites existing epics.md instead of appending new epics," "overwrites existing sprint-status.yaml instead of merging new stories"). This is a structural gap, not a bug — it recurs at the "living document" layer regardless of framework philosophy (spec-first vs agent-persona-first vs skill-first).
2. Install-time failure is astonishingly universal and almost always OS-specific (Windows). Four of five repos have a *currently open or recently closed* Windows-only installer crash: SuperClaude (#409 unhandled TypeError, #559 cp1252 UnicodeEncodeError), BMAD (#1149 TUI freeze, #1214 ESM require crash), ECC (#1435 GateGuard renameSync EEXIST, multiple Windows/Git-Bash issues dated as recently as 2026-07-09), superpowers (#1918 parenthesis-in-path SessionStart failure, #871 Cursor hooks-cursor.json). None of these are exotic edge cases — they are "first command a brand-new Windows user runs" failures, and they recur release after release rather than getting systematically regression-tested.
3. The "no uninstall command" complaint in SuperClaude isn't a one-off — it's the same ask filed four separate times (#397, #476, #485, #500) over roughly four months, and as of the most recent recent-issues pull it's still open. A missing uninstall path is a small thing engineering-wise but it is a trust signal: users who want a clean way *out* before they've even decided to stay are a leading indicator of low confidence in the install process itself.
4. The clearest, most articulate skepticism toward "measured rigor" workflow frameworks came not from a hater but from a self-described "FAN of BMAD" (issue #2003) who ran an actual retrospective with the BMAD agents themselves and concluded the human-supervision requirement is self-defeating: technical users can't review AI-scale code changes either, so the framework's core "human filters AI output" assumption breaks for both the technical and non-technical audiences it claims to serve. This is a more dangerous critique than a drive-by "too complicated" comment because it comes with receipts (a Mermaid diagram of the failure loop, specific epic/story examples) and ends with a concrete ROI number: 10-15x time cost for equivalent-or-worse quality on small projects.
5. Token/context cost complaints have shifted vocabulary between repos and over time. Early 2025 SuperClaude issues frame it as "How does CC load config files, do we have a 70k token issue?" (a curious question). By mid-2026, ECC issue #2482 frames the identical failure mode as a quantified, closed-with-a-fix engineering bug: "12.8k tokens injected into every single turn, ~90% unused." The framing has matured from "I wonder if this is a problem" to "here is the exact per-turn tax and here is the mechanism," which suggests the pain is real and growing, not a passing worry.
6. superpowers issue #743 ("Im seeing slowness in responses since using the skill") is the richest single thread I found for both confirmed pain AND abandonment signal in one place — 20 comments spanning March through late April 2026, escalating from "is this normal?" to specific comparisons ("35 minutes vs. 2 minutes with plain claude code," "20 minutes to write 2 loc using 75% of my 4-hour cc limit") to explicit abandonment ("I'm removing all this from my Codex now"). The maintainer (obra) engaged early asking for specifics but the thread kept accumulating "same here" reports for over a month with no visible fix landing in that window — a slow-burn credibility problem rather than a single sharp complaint.
7. A genuinely unexpected signal: two separate antivirus false-positive reports (ECC #2428 "ESET flags a SKILL.md as GenAISkill.IC Trojan," superpowers #1950 "is It contain Virus?!") — both filed within days of each other in early July 2026. This is a distribution-layer risk category I hadn't considered for a markdown-based skill/agent kit: shipping enough skill files with enough shell-invocation patterns inside them apparently now trips heuristic AV scanners, and users react by deleting the files or asking if the whole repo is malware, which is corrosive to trust regardless of whether it's a false positive.
8. The "overly-specified plans defeat the point of delegating to a model" complaint (superpowers #895) triggered a genuinely interesting maintainer-vs-user debate in the comments, with a user linking actual research ("AGENTIF: Benchmarking Instruction Following of LLMs in Agentic Scenarios") arguing over-specified instructions measurably hinder agent performance, while the maintainer argued the over-specification is intentional (to allow weaker/cheaper models to execute). Neither side conceded — this is a live, contested design tension in the "workflow kit" space, not a solved problem, and it maps directly onto any competitor (including us) that generates detailed plans/rules files for agents to follow.
9. BMAD's #1728 (global install / symlink request, 16 reactions) reveals a pain point that's almost invisible until you hit it: the standard "copy 4MB of framework files into every project" install pattern actively punishes users for upgrading, because the upgrade path deletes and recreates the framework directory, destroying any project-specific customization (custom agents, edited rules) that lived inside it. This is a design-pattern risk relevant to any kit (like ours) that writes generated files directly into a target repo rather than referencing a versioned, linkable source.
10. Cross-tool-support requests (Zed, Warp, Cursor, OpenCode, Copilot, Antigravity, Gemini CLI, Kiro, Trae, oh-my-pi, GitLab Duo...) dominate BMAD's and superpowers' top-reacted lists almost as much as bugs do. This tells me users treat "which coding agent/IDE do you support" as a first-class, continuously-relitigated cost of adoption — every new agentic CLI that ships seems to generate a fresh feature-request issue within weeks, and maintainers visibly struggle to keep pace (superpowers has *removed* Gemini CLI and Codex-hook support at points, per #1954 "Restore Gemini CLI support" and #1875/#1867 Codex hook still broken on 6.1.0 — i.e., regressions in tool support, not just gaps).
11. Genuinely surprising overlap in vocabulary: multiple unrelated users across different repos independently reached for the phrase "overengineering for trivial tasks" / "overkill" / "boil the ocean" / "AI slop" to describe the same underlying complaint — that the framework's process ceremony scales with the framework's ambition, not with the size of the actual task in front of the user. BMAD #2538 ("Excessive noise in generated code... AI slop") is the same complaint as superpowers #743/#1900 and ECC #2398, just applied to a different artifact (code comments vs. response latency vs. skill count). This looks like a category-level failure mode for "comprehensive AI workflow framework" products generally, worth taking seriously rather than dismissing as isolated grumbling.
