## Disconfirming evidence  (required)
- **Skepticism toward the DX-teardown genre itself.** On the HN thread discussing Tessa Kriesel's "3 minutes" teardown (2 points, 2 comments), commenter benoau called it "A contrived error-ridden, disastrous SaaS with barely functional registration, broken features, requiring invasive permissions and with no customer testimonials. But good news — their SaaS, which they neglect to link to in their article — can help you detect these discrete retention issues." https://news.ycombinator.com/item?id=46560590 (2026, exact month unclear from item metadata) — a reminder that "first-run UX teardown" content is frequently itself a lead-gen funnel, not disinterested research. Relevant self-check for a harness-branded launch.
- **"Don't rewrite the tool for agents — just stop being hostile."** On the 163-point HN thread about CLI design for agents, devmor: "If AI agents are so underdeveloped and useless that they can't parse out CLI flags, then the answer is not to rewrite the CLI." mahendra0203: "Agents are actually good at parsing messy human output... The real insight shouldn't be 'rewrite for agents'—it's 'make your CLI not actively hostile.'" https://news.ycombinator.com/item?id=47252459 (2026) — direct pushback against building elaborate new scaffolding/ceremony in the name of agent-compatibility.
- **"I don't see major differences between all of these systems."** Robert Hafner, on the proliferation of AI coding-assistant frameworks: "I really don't see major differences between all of these systems... I really do suggest experimenting with the latest options when you get started." He praises skipping frameworks entirely: "My personal cheat code for this is to never start from an empty repository... The scaffolding you get from these templates is a huge boost." https://blog.tedivm.com/guides/2026/03/beyond-the-vibes-coding-assistants-and-agents/ (2026-03) — a practitioner explicitly preferring plain project templates over any named methodology/workflow-kit.
- **Heavier workflow ceremony measurably costs real time, not just patience.** The same CRM-dashboard task took "12 minutes with OpenSpec, 90 minutes with Spec Kit, and 5.5 hours with BMAD" (https://reenbit.com/bmad-vs-spec-kit-vs-openspec-choosing-your-spec-driven-ai-framework/, 2026 Q1), and independently, BMAD Full was described as "Twelve agents, a heavy artifact set and a steep learning curve" against OpenSpec's "Lowest barrier to start... Best out-of-the-box experience" (https://ranthebuilder.cloud/blog/i-tested-three-spec-driven-ai-tools-here-s-my-honest-take/, 2026-04-13). This is the sharpest available evidence that a "measured harness" / multi-agent workflow-kit can be the *slower* path for a dev who just wants the task done — exactly the audience most likely to bounce off a rules/skills/agents-heavy kit in the first 10 minutes.
- **Optimizing purely for time-to-hello-world is itself criticized.** Scale Factory: "Is someone championing these for your platform, or are you stopping once you see 'Hello, World!' on the console?" — arguing MySQL and MongoDB won early adoption via short MTTHW (unauthenticated superuser access, no default security) at the cost of the 2017 ransomware wave against exposed databases. https://scalefactory.com/blog/2021/03/03/optimising-for-mean-time-to-hello-world-considered-harmful/ (2021-03, older but the argument is evergreen and directly cuts against a "TTFHW at all costs" first-run design).
- **People explicitly want the plumbing removed, not documented better.** The SuperClaude reporter didn't ask for clearer install docs — they asked to bring back the old one-command installer: "Maybe put that back so people not familiar with python and whats going on can still install it." https://github.com/NomenAK/SuperClaude/issues/146 (2025) — a vote for fewer moving parts over better-explained moving parts.
- **"I stopped using Copilot and didn't notice a decrease in productivity"** — attributed by Faros.ai to a recurring Reddit sentiment ("echoed repeatedly across the platform"), challenging the base assumption that any AI dev-workflow tool is even net-positive once the novelty wears off. https://www.faros.ai/blog/best-ai-coding-agents-2026 (2025-12-12 last updated; original Reddit thread not independently located, so treat this specific attribution as secondary-sourced).

## What surprised me
The clearest surprise is how rarely the actual "first 10 minutes" failure is a capability problem — across every source here, the thing that kills a first run is trust and legibility, not intelligence. Tessa Kriesel's teardown nails this: developers decided within 3 minutes based on unclear value prop, broken auth, unexplained errors, invasive permission asks, generic visual design, and absent social proof — none of which touch whether the underlying product actually works. That maps directly onto AI-dev-workflow kits: a wall of rules files and unexplained generated agents/skills reads exactly like "invasive permissions" and "generic design" to a skeptical first-time user, even if the generated output is excellent.

Second surprise: the AI-agent-specific tooling ecosystem (SuperClaude, BMAD-METHOD, ECC) is reproducing decades-old CLI packaging pain almost exactly — externally-managed-environment errors, pipx vs pip vs system-Python invocation mismatches, and contradictory install docs. These are not novel-to-AI problems; they are the same Python-packaging swamp that has burned developers since virtualenv existed, just now gating access to an agent framework instead of a web framework. A launch that ships a single, tested install path (and explicitly warns against combining install methods, the way ECC's README now does after apparently getting burned by exactly that) would already be ahead of three real, still-open issues I read.

Third surprise: the quantified cost of ceremony is larger than I expected and comes from sources with no stake in workflow-kit skepticism — two independent hands-on comparisons of BMAD vs. Spec Kit vs. OpenSpec, both landing on "heaviest process = slowest outcome" for the same real task (5.5 hours vs. 12 minutes, a 27x gap). This is a direct, discomfiting data point for anything positioning itself as a "harness" or "measured-outcomes" workflow generator: the market has independent, recent (April 2026), hands-on evidence that heavier structure can cost more wall-clock time than it saves, at least on a single realistic task. That doesn't mean structure never pays off — the same reviewers still liked BMAD Full's artifact depth for planning — but it reframes the pitch: a first-run must show the payoff fast, or it will be lumped in with BMAD's "steep learning curve, heavy artifact set" reputation before it gets credit for its depth.

Fourth surprise: praised onboardings (Stripe docs, Devin, "never start from an empty repo") share a narrow, repeatable set of traits and none of them is "asks good questions up front." Stripe's most-cited feature is that docs auto-fill *your own* sandbox data into code samples — it removes a step rather than adding a guided one. Devin's praise ("best UX I've had") sits right next to real confusion (looping on a wrong system date, editing a file back-and-forth) in the same review — meaning "impressive on the happy path" and "confusing on the very first edge case" are not mutually exclusive, and a launch should assume both will be reported by the same reviewer.

Fifth surprise, and possibly the most load-bearing for a workflow-kit specifically: nobody in this research asked for a better-explained wizard. The BMAD beginners in discussion #113 didn't want more docs, they wanted an in-flow "Am I doing the right thing?" checkpoint. The SuperClaude installer reporter didn't want better install docs, they wanted the old one-liner back. The general onboarding literature agrees: smart defaults and progressive disclosure beat more complete instructions, because completeness still costs the reader time and judgment they don't yet have context to spend. The implication for a rules/skills/agents-generating kit is specific: don't respond to "users are confused about setup" by writing a longer README — respond by removing steps or adding a single "you are on track" signal mid-flow.

Explicit answer — a 10-minute first run MUST contain: (1) a working, single install path with no branching decision the user has to get right on the first try; (2) a visible first win inside that window — generated output the user can see/run/diff before being asked to trust anything further; (3) errors that state what happened and what to do next, every time, no silent failures; (4) a plain statement of what access/permissions are being requested and why, before requesting them; (5) at least one in-flow "you're on track" signal for any multi-step flow, so users aren't guessing whether they've missed something; (6) a visible cost/scope estimate before any expensive or irreversible action runs.

It MUST NOT: (1) present install-method choices whose combination silently breaks (ECC's exact failure mode); (2) require Python/venv/package-manager literacy as a precondition for a tool that isn't a packaging tool; (3) generate a large, unexplained set of new files/agents/rules with no map of what each one is for — that reads as "generic/weekend-project" regardless of underlying quality; (4) rely on a lengthy README instead of an in-product checkpoint to resolve "am I doing this right"; (5) let a first-run failure surface as a bare stack trace or silent no-op with zero next step; (6) assume the value proposition is self-evident from a homepage a developer will read at most twice before deciding.
